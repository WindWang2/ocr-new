# YOLO 多目标检测与自动裁剪升级技术文档

## 1. 升级背景
原系统采用相机位（F0-F8）与仪器类型的强绑定模式，导致一个相机只能对应一台仪器的局限性。随着实验室布局的变化，单张全景大图中往往包含多个仪表。为此，我们引入了基于 **YOLO26x** 的多目标检测方案，实现了“全图扫描、自动抠图、精准读取”的闭环。

## 2. 核心技术组件

### 2.1 YOLO26x 旗舰引擎
- **模型版本**：YOLO26x (Best pt)。
- **分类定义**：
  - 类别 0-8：分别对应原有的 F0-F8 仪器 Prompt 模板。
- **手动 NMS 算法**：
  - **背景**：端到端（E2E）模型在小样本下易产生重复框，且其内部禁用了传统 NMS。
  - **解决方案**：在 `yolo_detector.py` 中强制引入 `torchvision.ops.nms`，设置 `iou_threshold=0.15` 和 `agnostic_nms=True`（跨类别抑制）。
  - **效果**：彻底解决了同一个仪表被画出多个重叠框的问题。

### 2.2 自动裁剪与存储流 (Auto-Crop)
- **逻辑实现**：在 `instrument_reader.py` 中，YOLO 检测到的每个 bbox 都会被增加 15px 的 Padding 后进行裁剪。
- **存储路径**：裁剪图自动保存至 `camera_images/F{id}/crops/` 目录下。
- **命名规范**：`原图名_crop_类别ID_时间戳.jpg`，确保唯一性。

### 2.3 相机与仪器解耦架构
- **API 层面**：`/experiments/{exp_id}/run-test` 接口现在不再盲目读取全图。它会调用 YOLO 获取所有目标，并智能寻找包含所需读数键（reading_key）的最佳裁剪图。
- **前端展示**：系统会将数据库中该条读数关联的图片路径更新为**裁剪后的特写图**。用户在前端看到的将是精准的仪表特写，极大地提升了用户体验和校验效率。

## 3. 运行环境要求
- **Python 环境**：`dfine` conda 环境。
- **关键依赖**：
  - `ultralytics`：提供 YOLO 模型支持。
  - `torchvision`：用于执行手动 NMS 算子。
  - `PIL (Pillow)`：处理图像裁剪与转换。
- **模型权重**：`/home/kevin/projects/ocr-new/models/yolo_instrument.pt`。

## 4. 后续优化方向
- **分类 Prompt 动态化**：目前类别 0-8 依然映射到固定 Prompt。未来可根据 YOLO 识别出的分类 ID，动态从数据库中拉取匹配的仪器识别模板。
- **多目标并发读取**：目前的 API 采用串行读取多个目标，未来可利用异步并行调用 LLM 以提升速度。
